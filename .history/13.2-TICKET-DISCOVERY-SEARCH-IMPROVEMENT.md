# 13.2-TICKET-DISCOVERY-SEARCH-IMPROVEMENT — Backlog Redesign: Two-Column Layout, Card List & Multi-Value Filters

## Ce que cette branche fait

Cette branche redessine entièrement la page Backlog : le tableau plat (`eui-table`) est remplacé par un layout deux colonnes (fluid layout eUI) avec un panneau de filtres repliable à gauche et une liste de cartes (`eui-content-card`) à droite. Les filtres passent de `eui-toggle-group` mono-valeur à des groupes de checkboxes multi-valeurs (status, type, priorité). Un header de liste affiche le nombre de résultats, un dropdown de tri, et des chips de critères sélectionnés avec suppression individuelle et "Clear all". Le backend est étendu pour accepter des filtres multi-valeurs séparés par virgule (`?status=TO_DO,IN_PROGRESS`). Un bug de pagination (binding `[page]` manquant sur `eui-paginator`) est corrigé, et 5 tickets supplémentaires sont ajoutés au projet API Gateway pour tester la pagination.

- Les paramètres `q`, `_sort`, `_order`, `_page`, `_limit` restent inchangés.
- 8 tests d'intégration ajoutés (Jest + supertest).

Fichiers : `mock/app/routes/project_routes.js`, `mock/app/routes/project_routes.test.js`

### 2. Frontend — Modèles & service mis à jour (STORY-002)

- Mise à jour de `BacklogListParams` : les champs `status`, `type`, `priority` passent de valeur unique à `string` (valeurs séparées par virgule).
- 3 tests de service ajoutés pour valider le passage des paramètres multi-valeurs.

Fichiers : `src/app/core/project/project.models.ts`, `src/app/core/project/project.service.spec.ts`

### 3. Frontend — Layout deux colonnes & liste de cartes (STORY-003)

- Remplacement complet du `eui-table` par un fluid layout eUI : `eui-page` > `eui-page-content` > `eui-page-columns` > `eui-page-column[hasSubColumns]`.
- Le page header est placé dans `eui-page-column-header-body` avec le bouton Create dans `eui-page-header-action-items`.
- Colonnes imbriquées (`eui-page-columns`) : colonne filtre repliable à gauche (`[isCollapsible]="true"`, `euiSize2XL`, `[isAutocloseOnMobile]="true"`) + colonne résultats à droite.
- Chaque ticket est rendu en `eui-content-card` avec sous-composants : `header-start` (type chip), `header-title` (titre), `header-subtitle` (clé ticket), `header-end` (status chip), `header-metadata` (priority chip coloré + assignee), `body` (description tronquée à 120 chars).
- Chips de priorité colorés : `eui-chip--danger` pour CRITICAL, `eui-chip--warning` pour HIGH, `eui-chip--info` pour MEDIUM.
- `eui-progress-bar[isIndeterminate]` pour l'état de chargement (pas de `eui-spinner` dans eUI).

Fichiers : `src/app/features/projects/backlog/backlog.component.ts`, `src/app/features/projects/backlog/backlog.component.html`, `src/app/features/projects/backlog/backlog.component.scss`

### 4. Frontend — Panneau de filtres checkbox (STORY-004)

- Recherche : `input[euiInputText]` avec debounce 300ms via `searchSubject`.
- 3 sections de filtres dans des `eui-card[euiCollapsible][euiNoShadow]` : Status (4 checkboxes), Type (4 checkboxes), Priority (4 checkboxes).
- Chaque section utilise `<fieldset>` + `<legend class="eui-u-sr-only">` pour l'accessibilité.
- Checkboxes `input[euiInputCheckBox]` avec `[(ngModel)]` sur des objets `statusChecks`, `typeChecks`, `priorityChecks`.
- Les valeurs cochées sont jointes par virgule et envoyées comme paramètre multi-valeur (ex: `status=TO_DO,IN_PROGRESS`).
- Tout changement de filtre remet la page à 1.

Fichiers : `src/app/features/projects/backlog/backlog.component.ts`, `src/app/features/projects/backlog/backlog.component.html`

### 5. Frontend — Header de liste, tri & chips de critères (STORY-005)

- Header de liste : compteur de résultats (`aria-live="polite"`) + dropdown de tri (`<select euiSelect>`) avec 6 options (ticket # desc/asc, title asc/desc, priority, status).
- Chips de critères sélectionnés : `eui-chip[isChipRemovable]` pour chaque filtre actif (recherche, status, type, priorité).
- Suppression d'un chip → décoche le checkbox correspondant et relance la requête.
- Bouton "Clear all" qui réinitialise tous les filtres, la recherche et le tri.
- Getter `activeFilterChips` qui construit dynamiquement la liste des chips à partir de l'état des filtres.

Fichiers : `src/app/features/projects/backlog/backlog.component.ts`, `src/app/features/projects/backlog/backlog.component.html`

### 6. Frontend — États vides & polish (STORY-006)

- Messages d'état vide contextuels : recherche sans résultat, filtres sans résultat, backlog vide par défaut, erreur de chargement.
- Hint affiché uniquement pour l'état vide par défaut (pas de filtres actifs).
- Bouton "Retry" affiché en cas d'erreur, avec growl notification via `EuiGrowlService`.
- Élément `<output>` pour l'état vide (sémantique HTML).
- 27 clés i18n ajoutées en EN et FR.

Fichiers : `src/app/features/projects/backlog/backlog.component.ts`, `src/app/features/projects/backlog/backlog.component.html`, `src/assets/i18n/en.json`, `src/assets/i18n/fr.json`

### 7. Bugfix — Pagination non fonctionnelle (BUG)

- Le `eui-paginator` n'avait pas de binding `[page]`, donc il restait toujours sur la page 0 visuellement même quand le composant envoyait `_page: 2` au backend.
- Fix : ajout de `[page]="(params._page ?? 1) - 1"` pour synchroniser l'état zero-indexed du paginator avec le `_page` 1-indexed du composant.
- Ajout de 5 tickets au projet API Gateway (tickets #11-#15) pour tester la pagination avec plus de 10 items.

Fichiers : `src/app/features/projects/backlog/backlog.component.html`, `mock/db/db.json`

## Décisions techniques clés

- **Fluid layout eUI plutôt que CSS custom** : utilisation du pattern `eui-page-columns` > `eui-page-column` natif d'eUI pour le layout deux colonnes. Cela donne gratuitement le repliage, l'auto-close mobile, et les labels verticaux. Pas de flexbox custom ni de `<aside>`.
- **Checkboxes multi-valeurs plutôt que toggle groups** : les `eui-toggle-group` ne supportent qu'une seule valeur sélectionnée. Les checkboxes `input[euiInputCheckBox]` permettent de sélectionner plusieurs valeurs simultanément (ex: TO_DO + IN_PROGRESS).
- **Cards plutôt que table** : `eui-content-card` offre plus d'espace pour afficher type, titre, clé, status, priorité, assignee et description dans un format scannable. Meilleure UX pour la découverte de tickets.
- **Chips de critères avec suppression** : `eui-chip[isChipRemovable]` rend l'état des filtres visible et actionnable. L'utilisateur voit exactement quels filtres sont actifs et peut les supprimer individuellement.
- **ResizeObserver mock dans les tests** : `EuiPageColumnsComponent` utilise `ResizeObserver` en interne, absent dans jsdom/vitest. Un mock global est ajouté via `beforeAll()` dans le spec file.
- **Binding `[page]` sur eui-paginator** : sans ce binding, le paginator ne se synchronise pas avec l'état du composant. Le paginator est zero-indexed, le backend est 1-indexed → conversion `(params._page ?? 1) - 1`.
- **`paginatorReady` guard** : `eui-paginator` émet `pageChange` à l'initialisation. Le flag `AfterViewInit` + `paginatorReady` empêche les appels API parasites au chargement.
- **`<output>` pour l'état vide** : élément HTML sémantique pour les résultats calculés, plutôt que `role="status"` sur un `<div>`.
- **`<section>` pour les chips** : élément sémantique avec `aria-label` plutôt que `role="region"` sur un `<div>`.
