## Key points 

### Angular 19

Angular new era is starting now, standalone by default, stable signals implementation, and many more improvements.

https://blog.angular.dev/meet-angular-v19-7b29dfd05b84


### eUI-ECL - sub-entries implemented

In v 19.x of eUI-ECL, it is possible to import components as sub-entries.

#### Migration Steps

In the showcase of each component, there is an example how to import the module for it. Example:
````
import { EclAccordionModule } from '@eui/ecl/components/ecl-accordion';
````
Importing all components with <code>EclAllModule</code> is still an available option.

#### Benefits
 - Smaller bundle size
 - Better performance


### NgRx removal from eUI internal usage / dependencies

It has been removed from eUI's internal architecture and dependencies list. This change streamlines the library's state management approach by transitioning to a lightweight Signal-based implementation.

#### Impact

- eUI no longer includes NgRx as a peer dependency
- Internal state management now uses Angular Signals
- Existing applications using NgRx through eUI will need to declare these dependencies explicitly

#### Migration Steps

-  If your application relies on NgRx through eUI, add the following dependencies to your `package.json`:

````
{ 
	"dependencies": { 
		"@ngrx/store": "latest-compatible-version", 
		"@ngrx/effects": "latest-compatible-version"
	}
}
````

-  Follow the official [NgRx installation guide](https://ngrx.io/guide/store/install) for detailed setup instructions

#### Benefits

- Reduced bundle size
- Improved performance through fine-grained DOM updates using Signals
- Simplified state management architecture
- Better alignment with Angular's native features

Need further assistance with migration? Contact our support team or refer to our migration guide.


### eUI emulated components style as default

For most of the components, we've enabled the emulated mode to avoid styles bleeding out of their host boundaries, something that was pending on our side since a while, but required lot of re-testing the entire library.

In terms of impact, it should be very smooth if you don't do much customization at your component level over eUI (it's anyway not recommended).

In the future releases, we'll go for more and more isolation of the component styles, the shadowDOM way / WebComponents way, research will be conducted in the course of v19 release.


### Release cycle updated to yearly release cycle

<em>Pending final decision from eUI steering board that will be conducted in Q1/2025.</em>

Following the recent survey we did, and also receiving feedback by lots of team within the MyWorkplace ecosystem / F&T portal and other big apps using eUI, time had come to reflect on the release cycle and the pressure it's putting on the development teams, on which we're perfectly aware since a couple of years, being impacted ourselves, having to maintain and follow Angular aggressive release cycle itself.
 
As v19 is the first "turning point" in the "new Angular" (Standalone by default, paving the way to the future of Angular (function based / less OoO driven / more adopting React style in their upcoming releases), we thought it was the time to review the release cycle to have more time between release for us / for you all to adapt your applications code to still be up-to-date in the future.
 
That's why, we will adapt the release cycle to a YEARLY release as of v19 release (due in early December), skipping an Angular version but still providing the same timespan of support : 18 months, as you can see in the chart below, the "active" part is just longer, "LTS" is reduced, in order to match the next v21 release of Angular in December 2025 : Yearly release will be done at the end of each year.
 
As a matter of fact, the amount of breaking changes (yearly) will be potentially bigger, but by experience working with Angular (2+) since the beginning we saw as well a lot of stability coming from the Angular team and their constant efforts to "make-it-easy" to upgrade, clear deprecation announcements, so the impact is reduced compared to early version of the framework, hence still trusting it to be the framework of choice for EC / eUI and for the future, specially developing webapps in an enterprise ecosystem like ours.
