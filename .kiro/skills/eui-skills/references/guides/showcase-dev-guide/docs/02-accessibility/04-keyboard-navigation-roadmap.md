# Keyboard navigation roadmap

## Overview
The aim of this page is to gather all the required information for a developer that wants to have proper keyboard navigation on the components she/he works in, answer to common questions raised when working on such tasks, have an alignment on the rules that need to be followed and clarify which patterns should be used per element based on the ARIA Authoring Practices Guide([APG](https://www.w3.org/WAI/ARIA/apg/)).

## Generic keyboard usage

* use the **tab** key to move forward and the **shift + tab** key to move backwards.
* use the **spacebar** or **enter** key to select/toggle an element. Enter key is the default one for selecting/toggling an element but there are elements(i.e. checkboxes and radiogroups) that are selected/toggled only through the spacebar.
* use the **arrow keys** to select/navigate on a radio group (left/right if it's a horizontal radio group, up/down if it is a vertical radio group), or use the defined keyboard access key with Alt + accesskey to select/toggle element.
* for **grouping widgets** such as menus, tablists, grids, or tree views, the parent element should be in the tab order (tabindex="0"), and each descendant choice/tab/cell/row should be removed from the tab order (tabindex="-1"). Users should be able to navigate the descendant elements using arrow keys.
* when a custom control becomes **disabled**, remove it from the tab order by setting tabindex="-1". Note that disabled items within a grouped widget (such as menu items in a menu) should remain navigable using arrow keys.


## Ensure that keyboard and mouse produce the same experience

To ensure that the user experience is consistent regardless of input device, keyboard and mouse event handlers should share code where appropriate. For example, the code that updates the tabindex or the styling when users navigate using the arrow keys should also be used by mouse click handlers to produce the same changes.

## Ensure that the keyboard can be used to activate element
To ensure that the keyboard can be used to activate elements, any handlers bound to mouse events should also be bound to keyboard events. For example, to ensure that the Enter key will activate an element, if you have an <code>onclick="doSomething()"</code>, you should bind doSomething() to the key down event as well: <code>onkeydown="return event.Code !== 'Enter' || doSomething();</code>".

## Prevent used key events from performing browser functions

If your widget handles a key event, prevent the browser from also handling it (for example, scrolling in response to the arrow keys) by using your event handler's return code. If your event handler returns false, the event will not be propagated beyond your handler. Another way is to use <code>event.preventDefault()</code> method.

## Always draw the focus for tabindex=-1 items and elements that receive focus programmatically
Choose between changing the background color to gray variations (i.e. disabled elements) or draw a focus outline(i.e. yellow) for elements  focused programmatically.

## How to set and track focus

* Use <code>element.focus()</code> to set focus:

    Do not use <code>createEvent(), initEvent() and dispatchEvent()</code> to send focus to an element. DOM focus events are considered informational only: generated by the system after something is focused, but not actually used to set focus. Use element.focus() instead.
* Use onfocus to track the current focus:

    Don't assume that all focus changes will come via key and mouse events: assistive technologies such as screen readers can set the focus to any focusable element. Track focus using <code>onfocus and onblur</code> instead.

    onfocus and onblur can now be used with every element. There is no standard DOM interface to get the current document focus. If you want to track the focus status, you can use the [document.activeElement](https://developer.mozilla.org/en-US/docs/Web/API/Document/activeElement) to get the active element. You can also use [document.hasFocus](https://developer.mozilla.org/en-US/docs/Web/API/Document/hasFocus) to make sure if the current document focus.

## preventDefault(), stopPropagation(), return false; on keyboard events

* **event.preventDefault()**: Prevents the browser default behavior, but does not stop the event from bubbling up the DOM. For example, when hitting 'Enter' on an <code>< a ></code> tag the default behavior is to navigate the browser to the location specified in the href attribute. If we set the <code>href</code> to #, this will cause the page to jump back to the top. Using <code>event.preventDefault()</code> we can prevent this default browser's behavior and still invoke the function attached to the anchor element.
* **event.stopPropagation**(): Prevents the event from bubbling up the DOM, but does not stop the browsers default behavior. When an event is called on an element, that event bubbles up the DOM and gets called on all of the elements parents. We can prevent this default bubbling behavior by using <code>event.stopPropagation()</code> so that the event is only registered by the element it is called upon. More info on event bubbling can be found [here](https://www.sitepoint.com/event-bubbling-javascript/).
* **event.stopImmediatePropagation()**: Throws the brakes on straight away, preventing even the siblings of the current listener from receiving the event.
* **return false**: Usually seen in jQuery, where it prevents the browsers default behavior, prevents the event from bubbling up the DOM, and immediately returns from any callback. Though in vanilla Javascript, returning false doesn’t have any effect on the default behaviour or event propagation of the element.

## Keyboard event listeners: @HostListener vs rxjs fromEvent

* **@HostListener**: Decorator that declares a DOM event to listen for, and provides a handler method to run when that event occurs. Angular invokes the supplied handler method when the host element emits the specified event, and updates the bound element with the result. If the handler method returns false, applies preventDefault on the bound element.
    * Gives you the added value of not having to manually trigger the change detection after an event is fired, which is something that needs to be done with Observables when a component uses the OnPush change detection strategy.
    * It does all the boilerplate of unsubscribing when the component is destroyed and as a result you don't have to manually unsubscribe.

* **fromEvent**: Creates an Observable that emits events of a specific type coming from the given event target.
    * Gives you more flexibility with execution and pipe lining as well as the possibility to bind outside the ngZone context.

<u>Comparison</u>:

@HostListener is very handy and the proposed technique when we want to listen to single events as it provides all the boilerplate for not subscribing/ unsubscribing to the events when the component is created/destroyed accordingly.

There are more advanced cases though where the use of @HostListener might be problematic and the use of rxjs fromEvent is recommended:

* <u>Missing Composability</u>: In cases where event composition is needed (i.e. when implementing a drag n drop feature and you need to combine mousemove, mousedown and mouseup events), all the event listeners will have to be registered for every component instance using the @HostListener although the aim is to register only one listener(i.e. mousedown) and when this event happens to register the other events accordingly. Not only it's complex to do all this logic within the event listener function but it's also difficult to disable a registered @HostListener function which will lead to constantly listen to events that are no needed and thus create performance issues. On the other hand rxjs fromEvent shines when it comes to composition of event-oriented code. The above mentioned scenario could be implemented as shown below without introducing performance issues or having to register/unregister event listeners (when using addEventListerner):

    <code>constructor(el: ElementRef) {

        const element = el.nativeElement;

        const mousedown$ = fromEvent(element, 'mousedown');

        const mousemove$ = fromEvent(document, 'mousemove');

        const mouseup$ = fromEvent(element, 'mouseup');
       

        mousedown$.pipe(switchMapTo(mousemove$.pipe(tap((event: any) => {
               this.style = {left: `${event.clientX}px`, top: `${event.clientY}px`};}),
            takeUntil(mouseup$))),takeUntil(this.destroy$)).subscribe();
    } 
    </code>
* <u>Lack of configuration options:</u> Following the [Angular documentation](https://angular.io/api/core/HostListener#description) regarding the @HostListener we notice that it is no possible to provide additional configuration options as the ones that we could pass when using addEventListener(capture, once, passive). Although these options are not used frequently, they could be very useful for heavily event-oriented features.

## Keyboard Navigable Lists using Angular CDK ListKeyManager

As mentioned in the Generic keyboard usage section when a component has nested children components that need to be focused the arrow keys should be used instead of the tab key to navigate. As lists are usually comprised of <code>< ul ></code> elements with nested <code>< li ></code> elements accordingly, the same rule applies regarding the usage of arrow keys. In general there are two techniques to manage keyboard navigation on lists: the **roving tabindex** and the **aria-activedescendant** attribute.

The **roving tabindex** technique sets all elements in the group to <code>tabindex="-1"</code> to be not reachable with a keyboard. Then it sets the selected element to <code>tabindex="0"</code> to be focus-able in the direct order  and listens for arrow keys keydown events. When there is a keydown event, it sets <code>tabindex="-1"</code> to the currently selected item and <code>tabindex="0"</code> to the newly selected item and calls focus on it.

The **aria-activedescendant** technique is based on the <code>aria-activedescendant</code> attribute which accepts an element id and announces the appropriate content when the selected element changes. Also, an element with <code>aria-activedescendant</code> attribute has to be focused. That technique also requires listening to arrow keys and moving the selected element manually. But the added value is that you need to update one parent element.

Based on these techniques, Angular CDK provides two varieties of **ListKeyManager**:  **ActiveDescendantKeyManager** and **FocusKeyManager**.

**ListKeyManager** is an event handler which calls the appropriate methods on the passed list items depending on a triggered event and manages the active option in a list of items based on keyboard interaction. Intended to be used with components that correspond to a <code>role="menu"</code> or <code>role="listbox"</code> pattern.

Each component that uses ListKeyManager will generally do three things:
* Create a <code>@ViewChildren</code> or <code>@ContentChildren</code> query for the options being managed.
* Initialize the ListKeyManager, passing in the options list.
* Forward keyboard events from the managed component to the ListKeyManager.

Meanwhile, each list item should implement the ListKeyManagerOption interface:

<code>
interface ListKeyManagerOption { 

disabled?: boolean;

getLabel?(): string;
 

}
</code>

Enable Wrapping: Navigation through options can be made to wrap via the <code>withWrap</code> method. i.e. <code>this.keyManager = new FocusKeyManager(...).withWrap();</code>

**ActiveDescendantKeyManager** is intended to be used with aria-activedescendant attribute. In that case, each list item should implement the Highlightable interface:

<code>
interface Highlightable extends ListKeyManagerOption {

  setActiveStyles(): void;

  setInactiveStyles(): void;

}
</code>

An example of using this technique within eUI can be found in the eui-dropdown component [here](https://citnet.tech.ec.europa.eu/CITnet/stash/projects/CSDR/repos/eui/browse/packages/components/eui-dropdown/eui-dropdown.component.ts).

**FocusKeyManager** is intended to be used when options will receive the browser focus directly. In that case, each list item has to implement the FocusableOption interface:

<code>
interface FocusableOption extends ListKeyManagerOption {

  focus(): void;

}
</code>

An example of using this technique within eUI can be found in eui-user-profile-menu component [here](https://citnet.tech.ec.europa.eu/CITnet/stash/projects/CSDR/repos/eui/browse/packages/components/layout/eui-user-profile/user-profile-menu/user-profile-menu.component.ts)

## Keydown, keypress or keyup?

* **keydown**: fires when any key is pressed down.
* **keypress**: It fires only when a key that produces a [character value](https://www.w3.org/TR/uievents/#character-value) is pressed down. For example, if you press the key a, this event will fire as the key a produces a character value of 97. On the other hand, this event will not fire when you press the shift key as it doesn't produce a character value. Moreover IE will not fire keypress events for non-alphanumeric keys(i.e Space).
* **keyup**: It fires when any key is released.

<u>Events firing order</u>: 1) keydown, 2) keypress, 3) keyup

<u>Comparison:</u> While both keydown and keyup events cover all the keys and are well supported by most browsers, there are a few differences that push keydown ahead of keyup. The keydown event fires before the browser processes the key, whereas the keyup event fires after the browser processes the key. If you cancel a keydown event (i.e. using event.preventDefault()), the browser's action will be cancelled too. In case of the keyup event, the browser's action will not be cancelled even when you have cancelled the event.
