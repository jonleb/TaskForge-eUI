# Accessibility (a11y) Steering Rules

This project follows the EU Web Accessibility Directive (2016/2102) and targets WCAG 2.2 Level AA conformance.

## General Principles

- Every UI implementation must be keyboard-navigable. Never rely solely on mouse/pointer interactions.
- Never override `outline: none` on focusable elements.
- Use semantic HTML elements (`<button>`, `<nav>`, `<main>`, `<table>`, `<th>`, `<td>`, `<label>`) over generic `<div>` or `<span>` with ARIA roles.
- All interactive elements must have a visible focus indicator.

## ARIA Attributes

- Add `aria-label` to icon-only buttons, links without visible text, and non-textual interactive elements.
- Use `aria-describedby` to associate help text or error messages with form controls.
- Use `aria-live="polite"` for dynamic content updates (e.g., growl notifications, table filter results count).
- Do not add redundant ARIA roles to semantic elements (e.g., no `role="button"` on `<button>`).

## Tables

- Always include a `<caption>` element describing the table's purpose.
- Use `scope="col"` on `<th>` elements in the header row.
- Use `data-col-label` on `<td>` elements for responsive table mode (eUI requirement).

## Forms

- Every form input must have an associated `<label>` (use `euiLabel` directive) with a matching `for`/`id` pair.
- Required fields must use the `euiRequired` directive or `aria-required="true"`.
- Validation errors must be announced via `aria-describedby` pointing to the error message element.
- Group related inputs with `<fieldset>` and `<legend>` (or `eui-fieldset`).

## Images and Icons

- Decorative icons: use `aria-hidden="true"`.
- Meaningful icons: provide `aria-label` on the icon or its parent interactive element.
- All `<img>` elements must have an `alt` attribute (empty `alt=""` for decorative images).

## Color and Contrast

- Never convey information through color alone. Use text labels, icons, or patterns as secondary indicators.
- Use eUI design tokens and utility classes (`eui-u-c-*`, `eui-u-c-bg-*`) which provide contrast-safe combinations.
- Boolean status columns (e.g., Active/Inactive) should display text ("Yes"/"No") not just color.

## Page Structure

- Each page must have exactly one `<h1>` (typically the `eui-page-header` label).
- Heading levels must not skip (e.g., no `<h1>` followed by `<h3>`).
- Use `eui-page` > `eui-page-header` > `eui-page-content` structure which provides proper landmark regions.

## Language

- The `lang` attribute must be set on the `<html>` element (handled by eUI's `I18nService`).
- Content in a different language than the page default should use the `lang` attribute on the containing element.

## Acceptance Criteria

Every feature implementation must include the following a11y checks in its acceptance criteria:

- [ ] All interactive elements are reachable and operable via keyboard (Tab, Enter, Space, Escape, Arrow keys)
- [ ] Screen reader announces content meaningfully (labels, roles, states)
- [ ] No color-only information conveyed
- [ ] Tables have `<caption>` and `scope="col"` on headers
- [ ] Form inputs have associated labels
- [ ] Focus order follows a logical reading sequence
